> 人类用户请阅读 [README.md](./README.md)。

## 仓库性质

本仓库是一个 **Obsidian 知识库**（非代码项目），用于管理和沉淀中文投资/交易领域的原始资料与 AI 加工内容。核心工作流是：**ASR 转录文稿 → AI 结构化总结 → Git 版本管理**。

- **仓库地址**：https://github.com/StormPhoenix/RawResourceBase.git
- **主分支**：`master`
- **文件格式**：Markdown（`.md`）为主，转录原始文件为纯文本（`.txt`）
- **编辑工具**：Obsidian（配置文件位于 `.obsidian/`）

## 核心工作流：直播转录 → AI 总结

### 必读文档

**在执行任何转录总结任务前，必须先完整阅读 [`直播视频文稿AI总结方法论.md`](./直播视频文稿AI总结方法论.md)。**

该文档定义了完整的处理流程，包括：

1. **前置判断**（第 1 章）：根据文件大小和标题关键词快速分档（A/B/C 类），决定输出粒度
2. **阅读与解析**（第 2.1 节）：ASR 字间空格格式的去噪策略
3. **知识提取**（第 2.2 节）：应提取 vs 应忽略的内容清单
4. **结构化输出**（第 2.3 节）：Frontmatter 模板 + 正文"倒金字塔"结构
5. **常见陷阱**（第 3 章）：假语义、口头禅误判、游戏隐喻等
6. **场景化策略**（第 4 章）：盯盘场 / 连麦答疑 / 专题讲座 / 游戏娱乐场
7. **质量自检清单**（第 7 章）：输出前逐项自查

### 处理流程

```
1. 读取 直播转录/ 下的 .txt 原始文件
2. 按方法论 1.1 节快速分档（文件大小 + 标题关键词）
3. 按方法论 2.1-2.2 节去噪、提取知识点
4. 按方法论 2.3 节生成结构化 .md 文件（含 Frontmatter）
5. 按方法论第 7 章执行质量自检
6. 将 .md 文件写入对应的 转录AI总结/ 目录
   - 文件名：与源 .txt 同名，仅扩展名改为 .md
7. （可选）Git 提交：源 txt 与生成的 md 一起提交
   - commit message 包含：BVID + 核心知识点梗概 + 处理深度(A/B/C)
```

### 文件命名约定

转录文件名遵循统一格式：

```
YY_MM_DD-HH-MM-SS_标题截断_BVID.txt
```

例如：`26_05_22-03-28-54_【个人观点】你们明天可能会吃肉，而不是割肉。_BV1PuLY6gETp.txt`

对应的 AI 总结文件名完全相同，仅扩展名为 `.md`。

### Frontmatter 规范（统一 YAML 格式）

本仓库所有 Markdown 文件统一使用 YAML Frontmatter，分为通用字段和转录专用字段两部分。

**模板：**

```yaml
---
# ── 通用字段（所有文件必填）──
title: 文章标题                    # 文章/视频标题
source_type: article              # article=外部转载 | transcript=直播转录
source_platform: 雪球              # 来源平台（雪球/知乎/B站/...）
source_url: https://xueqiu.com/...  # 原文链接
author: 行中衡                    # 作者/UP主
publish_date: 2024-08-04          # 发布日期（YYYY-MM-DD）

# ── 直播转录专用（仅 source_type=transcript 时填写）──
source_video: BV1xxx              # B 站视频 BVID
duration_estimate: ~N 分钟/小时    # 根据文本量估算
generated_at: YYYY-MM-DD          # AI 总结生成日期
has_finance_content: true/false   # 是否包含金融内容
---
```

**使用规则：**

- 外部转载文章（`source_type: article`）：只填通用字段，转录专用字段省略
- 直播转录总结（`source_type: transcript`）：填满全部字段
- `publish_date` 为内容原始发布日期；`generated_at` 为 AI 总结生成日期（仅转录类需要）
- Frontmatter 之后紧跟正文，正文以一级标题 `# 文章标题` 开头（标题保留到正文，便于 Obsidian 大纲与搜索）

**示例——外部转载文章：**

```yaml
---
title: 优势投资
source_type: article
source_platform: 雪球
source_url: https://xueqiu.com/1553799558/299829399
author: 行中衡
publish_date: 2024-08-04
---

# 优势投资

（正文原样保留……）
```

**示例——直播转录 AI 总结：**

```yaml
---
title: 豆区对有色的看法（直播回溯）
source_type: transcript
source_platform: B站
source_url: https://www.bilibili.com/video/BV16SML6YEAA
author: 做多中国的好人豆子
publish_date: 2026-07-09
source_video: BV16SML6YEAA
duration_estimate: ~5 分钟
generated_at: 2026-07-23
has_finance_content: true
---

# 豆区对有色的看法（直播回溯）

（正文……）
```

## 重要注意事项

### 内容处理原则

1. **保留人格色彩**：UP 主的口头禅、独特比喻、情绪表达是辨识度的来源，不要写成教科书
2. **区分内容与噪音**：60 分钟直播中真正的金融内容可能只有 5-10 分钟，游戏/闲聊内容应忽略
3. **跟踪知识演进**：同一 UP 主的连续直播中，关注观点一致性、认知进化、预判验证
4. **首次出现优先**：重复观点简写或引用前篇，首次出现的观点才完整记录
5. **区分前瞻预判与事后归因**：两者价值不同，须明确标注

### 外部文章归档规范

- 保存外部文章（知乎、雪球等）为 Markdown 时，使用统一 YAML Frontmatter（见上节模板），`source_type` 设为 `article`，转录专用字段省略。
- YAML Frontmatter 之后，正文以一级标题开头，保留原文章标题：`# 文章标题`
- 外部文章文件命名使用日期前缀格式：`YYYY-MM-DD_文章标题.md`（日期为文章原始发布时间）。
  例如：`2024-08-04_优势投资.md`
- **原文保真**：从外部网页（知乎、雪球等）下载文章到本地时，必须**一字不改**地保留原文内容，不得删减、改写、摘要或重新排版。仅在开头添加 YAML Frontmatter，正文部分原样输出。

### 图片与附件规范

- 图片统一存放在 `Attachments/` 根目录（扁平结构，不再使用子目录分类）
- Markdown 中引用图片使用 **Obsidian WikiLink** 格式：`![[文件名.jpg]]`
  - ❌ 不要使用 Markdown 链接格式：`![alt](路径/文件名.jpg)`
- 图片文件名使用描述性前缀避免重名，例如 `毛泽东_img_00.jpg`、`先导智能_01.jpg`
- 外部文章中的在线图片（如知乎、雪球）可保留原始 URL，无需下载到本地

### 大文件处理策略

| 文件大小 | 阅读策略 | 预期产出 |
|----------|---------|---------|
| < 8 KB | 快读，多数为测试/极短场 | A 类：1-2 行说明（~300-500 字节） |
| 8-40 KB | 中等阅读 | B 类：2-5 个知识点（~1-3 KB） |
| 40-80 KB | 深度阅读 | C 类：6-12 个知识点（~5-15 KB） |
| > 80 KB | 分段处理/读关键段落 | C 类++：分主题结构（~10-15 KB） |

### 禁止操作

- ❌ 不要修改 `直播转录/` 目录下的原始 `.txt` 文件
- ❌ 不要修改 `访谈/`、`毛泽东/`、`Sakin/` 下的文章内容
- ❌ 不要修改 `行中衡/` 下的文章正文内容（frontmatter 元数据维护除外）
- ❌ 不要修改 `Attachments/` 下的已有图片文件
- ❌ 不要修改 `.obsidian/` 目录下的配置文件
- ❌ 不要删除任何已有文件
- ❌ 不要在总结中编造原文没有的内容

### Git 操作规范

- 提交时将**源 txt 与生成的 md 一起提交**（如果是新转录文件）
- Commit message 格式建议：`add: BV1xxx - 核心知识点梗概 [C类]`
- 不要执行 `git push --force`
- 不要修改 `.gitignore`（如果存在）
